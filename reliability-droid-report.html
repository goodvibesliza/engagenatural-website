Alert Investigation Brand managers cannot open communities

SEV-2 Started: 2025-10-01 00:00:00 UTC
On-call: Droid (Factory)

Executive Summary

Brand managers could not open community views due to a broken route and a component import mismatch. We added a guarded brand route that reuses the staff community UI and fixed the component import; we also wired the brand CommunitiesManager “Open” action to the new route.

Root Cause Analysis

  • Primary Cause: No brand route existed to render the staff-style community page, and the “Open” action navigated to /community/:id which redirects/404s for brand managers.
  • Code Defect: EnhancedCommunityPage imported ./components/PostList.jsx but the actual file is PoslList.jsx, breaking the page if rendered.

Evidence

Code Reference

src/App.jsx – added route: /brand/community/:communityId guarded by RoleGuard(brand_manager)
src/components/community/EnhancedCommunityPage.jsx – fixed import to './components/PoslList.jsx'
src/components/brand/communities/CommunitiesManager.jsx – “Open” now navigates to /brand/community/:id

Proposed Fix

  1. Immediate Mitigation: Add brand route /brand/community/:communityId that renders the staff-style EnhancedCommunityPage under a brand_manager RoleGuard.
    Validation: Brand manager can open a community from CommunitiesManager without error.
  2. Short-term Fix: Update EnhancedCommunityPage import to match actual filename (PoslList.jsx), and change CommunitiesManager “Open” action to navigate to the new brand route.
    Validation: Page renders posts and UI elements as expected.
  3. Long-term Solution: Standardize component filename (“PostList.jsx”) and consider consolidating staff and brand community views behind a shared component to avoid drift.
    Validation: CI build + manual smoke test across staff/brand roles.
