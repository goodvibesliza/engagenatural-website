Alert Investigation – Staff notifications not visible after “Request Info”

SEV-2 Started: 2025-10-01 00:00:00 UTC
On-call: Droid (Factory)

Executive Summary

Staff users did not see system notifications created by the admin “Request Info” action due to missing Firestore rules on notifications/{uid}/system. We added precise owner-only rules (admins can create), verified real-time subscription on the staff Notifications page, and implemented an append-only Questions history on verification requests with backfill for legacy messages.

Root Cause Analysis

  • Primary Cause: Missing read permission for notifications/{uid}/system/* in firestore.rules blocked staff subscriptions from returning data.
  • Contributing: Admin UI overwrote a single infoRequestMessage instead of preserving an append-only Questions history.

Evidence

Code References

src/pages/admin/VerifyStaff.jsx – writes notifications/{uid}/system and now appends infoRequests via arrayUnion
src/pages/staff/dashboard/NotificationsPage.jsx – subscribes to notifications/{uid}/system; mark-as-read updates same doc
firestore.rules – new rules for notifications owner reads/updates; super_admin create

Proposed Fix

  1. Immediate Mitigation: Add Firestore rules for owner‑only reads/updates under notifications/{uid}/system; allow create by super_admin.
    Validation: Staff sees new items within ~2s and can mark as read; cross‑user reads are denied.
  2. Short‑term Fix: In VerifyStaff.jsx, stop overwriting infoRequestMessage; append to infoRequests via arrayUnion and show Questions history (latest first) with backfill for legacy.
    Validation: Two “Request Info” actions display two entries after reload.
  3. Long‑term: Optional one‑off migration to move legacy infoRequestMessage → infoRequests[0] and drop legacy fields.
    Validation: No requests rely on legacy fields.
